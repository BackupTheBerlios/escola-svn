Adaptação/Modificação

3.1

Naming fields trata da nomenclatura para nomear atributos de classes. O método da seção 3.1.1 deverá ser seguido, em detrimento daqueles das seções 3.1.1.1 e 3.1.1.2. O mesmo ocorre para a seção 3.1.2 Naming Components (Widgets), na qual as sugestões das seções 3.1.2.1 e 3.1.2.2 não deverão ser seguidas.

3.4.2.1

Apresenta um método que pode, em alguns casos, aumentar a dificuldade de manutenção do código. Em conseqüência, não deverá ser seguido.

4.1.2

Naming Loop Counters trata da escolha de nomes para variáveis contadoras. Esta seção nos oferece duas maneiras para declarar variáveis desta natureza. Nós optamos pelo método “i, j, k”, ou seja, para variáveis contadoras daremos preferência pelos nomes: i, j e k. Isto porque estes nomes já estão consagrados pelo uso.

5

tandard for Parameters (Arguments) To Member Funcitions mostra três alternativas para nomes de parâmetros (argumentos). Optou-se pelo método da seção 5.1 em detrimento daqueles das seções 5.1.1 e 5.1.2 e 5.1.3.

7.2

Use Wild Cards When Importing Classes mostra duas alternativas para importar classes. O conteúdo da seção 7.2.1 não será considerado pois apresenta um inconveniente: um número significativo de “imports” pode ser necessário, o que dificultaria a manutenção.

7.4

Wrintg Java Test Harnesses será descartada, pois a produção de código para teste deverá seguir outra abordagem, conforme documento Developement Case para o SISPG. A abordagem a ser seguida é o Junit (http://www.junit.org ).

9

Proposed javadoc Tags for Member Functions também será desconsiderada pois para documentação será utilizada a versão do javadoc contida no ambiente a ser utilizado. Esta definição está presente no documento Development Case.
